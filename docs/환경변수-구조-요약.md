# 환경 변수 구조 요약

Supabase·Notion 등 설정을 `.env.local` / `.env.production` / `.env.example`로 나눈 뒤, 저장소에 실제 키가 남지 않도록 정리한 내용입니다.

---

## 생성·수정된 파일

| 파일 | git 커밋 | 용도 |
|------|----------|------|
| `.env.local` | ❌ (gitignore) | 로컬 개발용 실제 값 |
| `.env.production` | ❌ (gitignore) | 로컬에서 `vite build` 등 프로덕션 모드 테스트 시 (빈 칸 채우기) |
| `.env.example` | ✅ 가능 | 팀 공유용 템플릿 (실제 시크릿 금지) |
| `.env` | ❌ (gitignore) | 공통 기본값만; 시크릿은 넣지 않음 |
| `.gitignore` | — | `.env.production`, `.env.*.local` 추가 |

---

## 변수 목록 (역할별)

### Supabase — 클라이언트 (`VITE_` 접두사, 브라우저 번들에 포함)

| 변수 | 로컬 | 프로덕션 |
|------|------|----------|
| `VITE_SUPABASE_URL` | `.env.local` | Vercel(또는 호스팅) 환경 변수 |
| `VITE_SUPABASE_ANON_KEY` | `.env.local` | 동일 |

anon 키는 설계상 클라이언트에 노출되는 값이지만, **URL·키 모두 코드에 하드코딩하지 않고** 환경 변수로만 둡니다.

### Supabase — CLI / CI (서버·도구 전용, 브라우저에 노출 금지)

| 변수 | 로컬 | 프로덕션 |
|------|------|----------|
| `SUPABASE_ACCESS_TOKEN` | `.env.local` | CI 시크릿 등 |
| `SUPABASE_PROJECT_REF` | `.env.local` | 동일 |

### Notion API (서버 전용, Vercel/Worker 런타임에서만 사용)

| 변수 | 로컬 | 프로덕션 |
|------|------|----------|
| `NOTION_API_KEY` | `.env.local` | Vercel 환경 변수 |
| `NOTION_DATABASE_ID_SECTIONS` | `.env.local` | 동일 |
| `NOTION_DATABASE_ID_TASKS` | `.env.local` | 동일 |
| `NOTION_DATABASE_ID_PLANTS` | `.env.local` | 동일 |

### API 기원 (프론트에서 Notion 프록시 호출 시)

| 변수 | 비고 |
|------|------|
| `VITE_API_ORIGIN` | 로컬은 Vite 프록시로 `/api` 처리 시 보통 불필요. GitHub Pages 등 정적 호스팅에서는 Vercel 도메인 등으로 설정. |

---

## 코드 쪽 참고

- `src/lib/supabase.js`는 `import.meta.env.VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`만 사용합니다. **소스에 URL·키 하드코딩 없음.**

---

## 새로 클론한 사람이 할 일

1. `.env.example`을 복사해 `.env.local`을 만들고 값을 채웁니다.
2. 프로덕션 배포 시에는 호스팅(Vercel 등)의 Environment Variables에 동일 이름으로 등록합니다.

자세한 배포 흐름은 `docs/바이브코딩/DEPLOYMENT.md`를 함께 보시면 됩니다.
